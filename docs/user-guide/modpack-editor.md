# Modpack Editor

The modpack editor is a built-in tool for creating and editing modpack JSON definition files (`installable.json`). It provides a structured graphical interface with validation, Modrinth search integration, diff tracking, and URL verification -- everything you need to author and maintain modpacks without editing raw JSON.

Access the modpack editor from the [Edit Mod Packs](edit-modpacks.md) screen.

## Toolbar

The toolbar across the top of the editor provides the core file and validation operations:

| Button | Description |
|---|---|
| **New** | Creates a blank modpack definition from a template. All required fields are pre-populated with placeholder values that you fill in. |
| **Open File** | Loads an existing `installable.json` definition from your local disk for editing. |
| **Open URL** | Fetches and loads a modpack definition from a remote URL. Useful for editing a modpack that is already published and hosted online. |
| **Save** | Writes the current definition to a local JSON file. You choose the save location. |
| **Validate** | Checks the entire definition for errors: missing required fields, malformed URLs, incorrect hash formats, and other structural problems. Results are displayed in a report. |
| **Diff** | Compares the current in-memory state of the definition against the last saved version, showing exactly what has changed. Useful for reviewing your edits before saving. |

## Metadata Fields

The top section of the editor contains the core modpack information:

| Field | Description |
|---|---|
| **Name** | The display name of the modpack, shown in the launcher's modpack list. |
| **Version** | The modpack version string (e.g., `1.0.0`). The launcher uses this to detect updates -- a newer version number on the server triggers an update badge. |
| **Description** | A text description of the modpack shown in the center panel of the main screen. |
| **Author** | The modpack author's name. |
| **Minecraft Version** | The Minecraft release version this modpack targets (e.g., `1.20.1`). |
| **Forge Version** | The Forge build version to install. Use the **Forge version picker** (see below) to browse available builds. |
| **Website URL** | Optional URL to the modpack's homepage, linked from the main screen's Website button. |
| **Logo URL(s)** | Optional image shown as the modpack's logo. Enter **one URL per line** -- the first line is the primary, and any additional lines are fallback/mirror URLs the launcher tries in order if an earlier one fails to download. A single URL is just one line. |
| **Background Image URL(s)** | Optional image displayed as the modpack's background on the main screen. Like the logo, accepts **one URL per line** (first = primary, rest = fallbacks). |

> **Manifest format:** `packLogoURL` and `packBackgroundURL` in `installable.json` accept either a single string (as before) **or** an array of strings. The launcher tries each URL in order and falls back to the bundled default if all fail; the optional `packLogoSha1` / `packBackgroundSha1` applies to whichever mirror resolves. Existing single-string manifests keep working unchanged.

### Forge Version Picker

The Forge version picker lets you browse all available Forge builds for the selected Minecraft version. Instead of manually looking up Forge version strings, click the picker to see a list of builds and select the one you want. The editor automatically populates the Forge version field with the correct value.

## File Tabs

Modpack files are organized into categorized tabs. Each tab manages a specific type of file:

| Tab | Description | Install Location |
|---|---|---|
| **Mods** | Forge mod JAR files. These are the core mods that define the modpack's gameplay. | `mods/` |
| **Configs** | Configuration files for mods. These control mod behavior and are typically generated by mods on first run, then customized by the modpack author. | `config/` |
| **Resources** | Resource packs and other game assets. | Varies (typically `resourcepacks/`) |
| **Shaders** | Shader packs for OptiFine or Iris. | `shaderpacks/` |
| **Initial Files** | Files that are copied only on the first install and never updated afterwards. Use this for default settings like `options.txt` that players may customize. | Varies |

### Managing File Entries

Within each tab, you can:

- **Add** a new file entry by providing a download URL and SHA-1 hash.
- **Remove** a file entry from the modpack.
- **Edit** an existing entry's URL, hash, or other properties.
- **Auto-calculate hashes** -- the editor can download the file from its URL and compute the SHA-1 checksum automatically, so you do not need to calculate hashes manually.

Each file entry includes:

| Property | Description |
|---|---|
| **File name** | The name of the file as it will appear on disk. |
| **Download URL** | The URL where the launcher will download this file from. |
| **SHA-1 Hash** | The expected SHA-1 hash of the file, used for integrity verification. |

## Modrinth Search

The editor includes integrated search for the [Modrinth](https://modrinth.com/) mod repository. Instead of manually finding download URLs and hashes for mods:

1. Open the Modrinth search panel.
2. Type a mod name to search.
3. Browse the results, which show mod names, descriptions, and available versions.
4. Select a result and choose the appropriate version for your Minecraft and Forge version.
5. The editor automatically populates the download URL and SHA-1 hash fields with the correct values from Modrinth.

This significantly speeds up the process of adding mods to a modpack, especially when building a pack with many mods.

## Version Bumping

The **Bump Version** button increments the modpack's version number. This is important for publishing updates:

- The launcher detects updates by comparing the version number in the hosted manifest against the locally installed version.
- Bumping the version before saving and re-uploading ensures that all users of your modpack will see the update badge and receive the new files on their next launch.

## URL Checking

The editor can verify that all file download URLs in the definition are **reachable** by making HTTP HEAD requests to each URL. This helps you catch:

- Broken links (404 errors) from moved or deleted files.
- Typos in URLs.
- Hosting issues before your users encounter download failures.

Run URL checking before publishing to ensure a smooth experience for your users.

## Workflow: Creating a New Modpack

A typical workflow for creating a new modpack from scratch:

1. Click **New** to start with a blank template.
2. Fill in the metadata fields: name, version, description, author, Minecraft version.
3. Use the **Forge version picker** to select the correct Forge build.
4. Switch to the **Mods** tab and add mods using **Modrinth search** or by entering URLs manually.
5. Add any config files, resource packs, or shaders in their respective tabs.
6. Click **Validate** to check for errors.
7. Click **Save** to write the definition to disk.
8. Upload the JSON file and all referenced files to your web server.
9. Share the URL with users so they can add the modpack via [Edit Mod Packs](edit-modpacks.md).

## Workflow: Updating an Existing Modpack

1. Click **Open URL** to load the currently published definition.
2. Make your changes (add/remove mods, update configs, etc.).
3. Click **Bump Version** to increment the version number.
4. Click **Diff** to review what changed.
5. Click **Validate** to check for errors.
6. Click **Save** and upload the updated file to your server.

> **Tip:** Use **Validate** before saving to catch issues early. The validator checks for required fields, valid URLs, and correct hash formats. Fixing problems in the editor is much easier than debugging them after publication.
